# ADR-004: Capability and Fallback Matrix

**Status:** Proposed

## Context
Target devices vary widely in performance. A single scene must scale from Tier 0 (Basline Compatibility) to Tier 3 (Full Spatial).

## Decision
We have implemented a **Capability & Fallback Matrix** based on `CapabilityTier` (0-3).

### Capability Matrix
| Feature | Tier 0 | Tier 1 | Tier 2 | Tier 3 |
| :--- | :--- | :--- | :--- | :--- |
| **Shadows** | None | Simple Blobs | Dynamic Soft | Full PCSS |
| **Fog** | Distance | Linear | Volumetric | Adv. Scattering |
| **Blur (Dialogue)** | Hide BG | Desaturate | Gaussian 3x3 | Gaussian 11x11 |
| **LOD** | Low-Aggression | Mid-Density | Full | Ultra-Density |

### Fallback Mechanism: **Tier-Filtered Schema Resolution**
All spatial assets must provide a `TierFallbackDeclaration`. The `TierDetector` runtime probes hardware capabilities (VRAM, Draw Call capacity) and assigns a tier that filters the incoming `SpatialMapOverlay`.

## Rationale
Using an explicit tier-based matrix allows authors to design a single scene and rely on the Presentation Core to automatically downscale resources or disable expensive passes (like volumetric fog) without manual per-platform scene edits.

## Consequences
- Every new visual feature must declare its tier assignment.
- Authors must specify fallback behavior for key visual elements.
