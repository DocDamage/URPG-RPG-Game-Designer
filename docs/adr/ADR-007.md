# ADR-007: Render Pass Architecture

**Status:** Proposed

## Context
A spatial renderer requires a structured execution order (Depth Prepass, Shadow Map, etc.) that the backend cannot improvise.

## Decision
We have implemented a **Tier-Aware Render Pass Architecture** (TARPA).

### Standard Execution Order
1.  **Z-Prepass** (Tier 2+): Depth-only draw calls to minimize overdraw.
2.  **Shadow Atlas** (Tier 1+): Projection of dynamic casters into a shared atlas.
3.  **Main Pass**: Depth-tested color pass using `ShaderPipelineState` hashes.
4.  **Post-Process Bloom/Fog**: Scene-wide atmosphere application.
5.  **Overlay UI**: Final compositing of UI/Dialogue elements.

### Pass Logic: **Immutable Hashing**
The `ShaderPipelineState` uses a unique identifier to minimize state changes. The Backend sorts `FrameIntent` commands by their pipeline hash before execution.

## Rationale
Using a deterministic pass order and hash-based sorting minimizes GPU pipeline stalls and state thrashing. This architecture is required to meet the 16.6ms frame time budget on Tier 1 hardware.

## Consequences
- Render command families are tied to pass assignment.
- Adding a new pass type becomes an architectural change.
