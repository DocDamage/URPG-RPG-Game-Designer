# ADR-009: Threading Model

**Status:** Proposed

## Context
Presentation Core must not introduce race conditions or hinder future multi-threading.

## Decision
We have established a **Strict Decoupled Threading Model** (SDTM).

### Thread Obligations
| Thread | Responsibility | Synchronization |
| :--- | :--- | :--- |
| **Game Thread** | Scene State Logic, User Input, World Update. | Reads `SceneState`. |
| **Presentation Thread** | Core Translation, `FrameIntent` Building. | Consumes `SceneState`, writes to `ArenaBuffer`. |
| **Render Thread** | Backend Execution, API Calls (Vulkan/DX12). | Consumes immutable `FrameIntent` from `ArenaBuffer`. |

### Immutable Intent Constraint
Once the `PresentationFrameIntent` is submitted by the translator via the `PresentationBridge`, its contents are **read-only** and its memory is managed by the `PresentationArenaAllocator`.

## Rationale
Decoupling the interpretation (Presentation) from the execution (Render) allows for parallelization without mutex overhead. Using a per-frame Arena ensures that zero heap allocations are performed in the hot loop.

## Consequences
- All public API calls must document their thread requirements.
- Violations are immediate code review blockers.
