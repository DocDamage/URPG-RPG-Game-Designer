# ADR-001: Presentation Core Ownership Model

**Status:** Proposed

## Context
Presentation logic is currently scattered across renderer, map code, and editor hacks. This leading to "drift" and makes supporting multiple presentation modes (Classic 2D vs Spatial) difficult and error-prone.

## Decision
The **Presentation Core** is established as the sole source of truth for visual interpretation.

### Ownership Model
| System | Responsibility | Constraints |
| :--- | :--- | :--- |
| **Presentation Core** | Scene Translation, Registry, LOD/Streaming, Tier Scaling. | Must be stateless; builds `PresentationFrameIntent`. |
| **Render Backend** | Pipeline Execution, Batching, Sorting, GPU Allocations. | Consumes `FrameIntent`; cannot access Game State. |
| **Scene Manager** | Lifecycle, State Logic, User Input Routing. | Cannot issue draw calls; provides `SceneState` to Core. |
| **Asset Pipeline** | Schema Validation, Metadata Ingestion, Fallback Generation. | Ensures all assets have a valid `TierFallbackDeclaration`. |
| **Editor** | Tooling, Live-Preview, Hot-Reload Signal. | Triggers `PresentationRegistry` refreshes. |

## Rationale
Centralizing visual intent allows the engine to switch between 2D Heritage, High-Fidelity Spatial, and Mixed-Mode presentation without modifying gameplay logic. This decoupling is essential for multi-platform scaling.

## Consequences
- Any presentation semantic in renderer code is a review blocker.
- Increased abstraction layer between game systems and the rendering backend.
