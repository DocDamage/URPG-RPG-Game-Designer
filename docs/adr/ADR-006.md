# ADR-006: Migration Policy

**Status:** Proposed

## Context
Legacy projects must be upgradable to the new Presentation Core model without losing data or faking completeness.

## Decision
We have established a **Non-Destructive Migration Policy** (NDMP) for the Presentation Core.

### Migration Stages
1.  **Inference (Auto)**: Map-level elevation is inferred from legacy tile types and existing collision flags; props are auto-anchored to the ground plane.
2.  **Decoration (Manual/Tweak)**: Users refine the generated `SpatialMapOverlay` (adjusting pitch, adds fog, etc.) via the Editor tools.
3.  **Upgrade (Final)**: The project's `ProjectPresentationSettings` is updated to a spatial-versioned schema.

### Reversibility Rules
-   **Side-by-Side**: The `SpatialMapOverlay` is stored as an auxiliary resource; the original legacy map data remains untouched for compatibility.
-   **Safe-Mode**: Core interprets spatial projects even if the overlay is accidentally deleted (defaults to Tier 0 fallback).

## Rationale
Using an auxiliary overlay approach prevents data loss while allowing users to move from "Flat 2D" to "Rich Spatial" incrementally. Legacy data is never modified—only augmented.

## Consequences
- Migration tooling must generate reviewable diffs.
- "Silent failure" during migration is a blocker.
