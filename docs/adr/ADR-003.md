# ADR-003: Spatial Data Representation

**Status:** Proposed

## Context
Spatial presentation requires structured data for elevation, props, lighting, and fog that "Classic 2D" does not natively handle.

## Decision
We have established a **Structured Map Overlay** for spatial data.

### Representation Format
| Component | Representation Type | Detail |
| :--- | :--- | :--- |
| **Elevation** | Grid-based (`ElevationGrid`) | Deterministic discrete steps (0.25 unit height increments). |
| **Props** | Reference-based (`PropInstance`) | Points to `AssetID` with local `Transform` and `InstanceData`. |
| **Lighting** | Array-based (`LightProfile`) | Supports Point, Spot, and Directional types. |
| **Fog/Atmosphere** | Global-Scene Block (`FogProfile`) | Distance-aware and volumetric data. |

### Coordinate System
Uses the **URPG Spatial Grid**:
- `X, Y`: Integer coordinate for the map tile.
- `Z`: Floating-point elevation for that tile.
- `Sub-Y`: Local offset within the tile for fine prop placement.

## Rationale
Using a grid-based elevation model with discrete steps ensures that pathfinding and spatial collisions can remain deterministic and high-performance, even on legacy (Tier 0) platforms. Prop referencing ensures that we maintain a single-source-of-truth for mesh and material assets.

## Consequences
- Schema choice determines asset pipeline requirements.
- Directly impacts the design of the Elevation Brush and Prop Placement tools.
