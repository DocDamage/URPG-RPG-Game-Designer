# ADR-008: Asset Pipeline Expectations

**Status:** Proposed

## Context
Spatial metadata (pivots, anchors, height class) becomes load-bearing in the new system.

## Decision
We have established a **Metadata Enforcement Contract** for all assets ingested into the Presentation Core.

### Asset Contract Requirements
| Asset Type | Required Metadata | Validation Rule |
| :--- | :--- | :--- |
| **Actor** | `PivotOffset`, `HeightClass` | Must have valid anchoring data for ground-snapping. |
| **Prop** | `AssetID`, `BBox` | Must have bounding box data for LOD/Culling. |
| **Material** | `TierFallback` | Must declare a valid shader variant for each `CapabilityTier`. |
| **Shader** | `HashState` | Must specify pipeline state (Blend, Depth, Culling). |

### Enforcement Point: **Registry Hydration**
The `PresentationRegistry` validates all metadata on cold-boot or hot-reload. Assets missing required metadata are flagged as "TBD-Visual" and assigned a magenta-checkerboard Tier 0 fallback.

## Rationale
Enforcing metadata at the registry level prevents the Presentation Core from interpreting invalid spatial data, which could otherwise lead to undefined behavior in the Render Backend.

## Consequences
- Classic-safe assets must continue working in basic modes without spatial claims.
- Build-time failures for assets claiming spatial compatibility but missing metadata.
